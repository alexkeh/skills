# GVA Skill Usage

This guide explains how to use the `oke-gva-deployer` skill to add a Generic VNIC Attachment (GVA) node pool to an OKE cluster.

## Quick Start (Recommended)
Run the guided menu script:

```bash
./scripts/gva-menu.sh
```

The menu:
- Uses only the **cluster name** you explicitly enter. There is no default cluster.
- Pulls cluster OCID from `~/.kube/config` if available.
- Always asks for the region and uses the value you reply with for the workflow.
- Auto-discovers cluster details (compartment, k8s version).
- Lists VCNs, then subnets, then OKE images for your k8s version.
- Prompts for fresh mutable node-pool inputs on each new create flow.
- Ignores old saved JSON payload files for new node-pool creation.
- Prints a ready-to-run OCI CLI command and a test Deployment manifest.
- Validates `ipCount` range and only lists discovered secondary subnets that are IPv4-only with more than one IPv4 CIDR block.

## Manual Flow (No Menu)
If you can’t run the menu, follow this sequence:

1. **Identify cluster OCID**
   - From kubeconfig: `~/.kube/config` contains `--cluster-id <ocid>` in the user exec args.
2. **Fetch cluster details**
   - `oci ce cluster get --cluster-id <cluster_ocid> --region <region>`
3. **List VCNs**
   - `oci network vcn list --compartment-id <compartment_ocid> --region <region>`
4. **Select a VCN and list subnets**
   - `oci network subnet list --compartment-id <compartment_ocid> --vcn-id <vcn_ocid> --region <region>`
5. **Pick OKE image for the cluster’s k8s version**
   - `oci ce node-pool-options get --node-pool-option-id <cluster_ocid> --region <region>`
6. **Create node pool with GVA**
   - Use the `oci ce node-pool create` command generated by the skill.
7. **Verify**
   - `kubectl describe node <node_name>` for GVA extended resources and taint.

This is the preferred fast-path discovery because it scopes queries to the selected VCN and avoids broad compartment-wide subnet discovery.

Constraint:
- Do not query existing node pools (`oci ce node-pool list/get`) as part of this skill flow.
- Build the new node pool from cluster metadata, VCN/subnet discovery, image options, and explicit user inputs.
- Never assume a cluster name.
- Always ask for the region and use the user-provided region for the current workflow.

## OCI CLI Support
GVA node-pool flags are expected in the regular OCI CLI. A preview OCI CLI is no longer required for this workflow.

```bash
oci --version
oci ce node-pool create --help | grep -E 'secondary-vnics|cni-type'
```

If the flags are missing, upgrade OCI CLI through your normal install path before running create. You can still print the generated command for review.

## Inputs You Still Need to Provide
Even with discovery, you must choose:
- Node pool name
- Node shape (and OCPUs/memory for Flex)
- Node count
- Node placement subnet (from VCN list)
- GVA VNIC profiles (subnet, applicationResource, ipCount)
- Each GVA secondary subnet must be IPv4-only and have more than one IPv4 CIDR block
- Image (from the menu list)

Always confirm additional GVA variables:
- NSG IDs per GVA profile
- Secondary VNIC display name (recommended)
- Whether additional profiles are needed
- Optional node-pool parameters (tags, labels, boot size, SSH key)

## Test Deployment
The menu prints a test Deployment manifest. Replace `<image>` with a real image and apply:

```bash
kubectl apply -f gva-test.yaml
```

Ensure the pod requests **exactly one** Application Resource and includes the required toleration.

## Troubleshooting
- **Pods Pending**: Check resource name, toleration, and IP capacity.
- **Validation errors**: Pods must request exactly 1 unit of a single Application Resource.
- **No extended resources on node**: GVA was not enabled on the node pool.
